### 9.1.2 网络隔离模拟策略

Gazebo 单机仿真无法真实模拟 WiFi，通过 CycloneDDS Domain 隔离实现逻辑分离：

```plain text
| Domain | 网络接口 | 用途 | 仿真方式 |
|--------|---------|------|---------|
| Domain 0 | lo (localhost) | 机器人内部通信 | localhost 模拟有线 |
| Domain 42 | lo (localhost) | fleet 协同通信 | localhost 模拟无线，IP 段 192.168.42.x |
```

domain_bridge 在每个 robot 进程中运行，桥接 Domain 0 和 Domain 42。

### 9.4 各场景详细验证方法

### 9.4.1 场景 A：走廊对向行驶（E2E-SIM-001）

前置条件：corridor.world，robot_a spawn @ x=-7 朝东，robot_b spawn @ x=+7 朝西。

操作步骤：

1. 启动仿真后观察 RViz：

```bash
rviz2 -d ~/ros2_ws/src/fleet_sim_gazebo/config/fleet_sim.rviz
```

1. 监控协调状态（robot_a 侧）：

```bash
ros2 topic echo /robot_a/fleet/coordinator_status
```

1. 监控 robot_b 侧：

```bash
ros2 topic echo /robot_b/fleet/coordinator_status
```

预期行为（robot_id 字典序：robot_a < robot_b，故 robot_a 优先）：

```plain text
| 时间 | robot_a 状态 | robot_b 状态 | robot_a 速度 | robot_b 速度 | 说明 |
|------|------------|------------|------------|------------|------|
| T+5s | AWARENESS | AWARENESS | 100% | 100% | 互相发现，距离>8m |
| T+~12s | CAUTION | CAUTION | 50% | 50% | 进入4m警戒区 |
| T+~15s | PASSING | YIELDING | 30% | 0% | 检测到路径冲突，协商完成 |
| T+~22s | NORMAL | NORMAL | 100% | 100% | robot_a 通过，双方恢复 |
```

Pass Criteria：

- ✅ robot_b 在距离 < 2.5m + 路径冲突时立即停车（YIELDING, 速度=0）

- ✅ robot_a 减速通过（PASSING, 速度=30%）

- ✅ robot_a 通过后发送 RESUME，robot_b 收到后恢复行驶

- ✅ 全程无碰撞（Gazebo 中模型未重叠）

- ✅ 让行延迟（从 YIELDING → NORMAL）≤ 15s

失败判定：

- ❌ 任意时刻两机器人碰撞

- ❌ robot_b 在 robot_a 通过后未恢复

- ❌ 让行延迟 > 15s

### 9.4.2 场景 B：同向追赶（E2E-SIM-002）

前置条件：corridor.world，robot_a spawn @ x=-10, robot_b spawn @ x=-3，两者同向（都朝 +x 方向）。

操作步骤：

1. 注入场景：

```bash
ros2 launch fleet_sim_gazebo inject_scenario.launch.py \
  scenario:=corridor_chase
```

1. 监控速度变化：

```bash
ros2 topic echo /robot_a/cmd_vel
ros2 topic echo /robot_b/cmd_vel
```

预期行为：

```plain text
| 时间 | robot_a 状态 | robot_b 状态 | 说明 |
|------|------------|------------|------|
| T+5s | NORMAL | NORMAL | 正常行驶，robot_a(后方)速度更快追赶 robot_b(前方) |
| T+~10s | CAUTION | NORMAL | robot_a 接近 robot_b，robot_a 减速至50%（路径平行无冲突，不触发让行）|
| T+~15s | CAUTION/NORMAL | NORMAL | robot_a 保持安全距离(≥2.5m)跟随，无让行协商 |
```

Pass Criteria：

- ✅ robot_a（后方追赶者）进入 CAUTION 减速至50%，不触发 YIELDING

- ✅ robot_b（前方被追者）不受影响，保持正常行驶

- ✅ 两车最小距离 ≥ 2.5m（YIELD_RANGE）

- ✅ 无死锁或不必要的停车

失败判定：

- ❌ 任一机器人触发 YIELDING 停车（同向路径平行，不应发生）

- ❌ 两车距离 < 2.5m

- ❌ 发生碰撞

### 9.4.3 场景 C：T字路口交汇（E2E-SIM-003）

前置条件：t_junction.world，robot_a @ x=-8, y=0 朝东，robot_b @ x=0, y=-8 朝北。

操作步骤：

```bash
ros2 launch fleet_sim_gazebo simulation_2robots.launch.py \
  world:=t_junction \
  scenario:=t_junction
```

预期行为：

两台机器人在进入路口前（距离交汇点 < 2.5m + 路径冲突检测）完成协商。

```plain text
| 排名 | 机器人 | 行为 |
|------|--------|------|
| 高优先级 | robot_a | 减速通过（PASSING 30%） |
| 低优先级 | robot_b | 停车等待（YIELDING 0%） |
```

Pass Criteria：

- ✅ 协商在两车进入交汇点前完成

- ✅ 无碰撞（任意时刻模型位置不重叠）

- ✅ 高优先级先通过

失败判定：

- ❌ 两车在路口内发生碰撞

- ❌ 协商过晚，来不及停车

### 9.4.4 场景 D：三方交汇排队（E2E-SIM-004）

前置条件：multi_robot_area.world，robot_a/b/c 分别在 180°/0°/120° 位置，同时向对向点行驶。

操作步骤：

```bash
ros2 launch fleet_sim_gazebo simulation_3robots.launch.py \
  world:=multi_robot_area \
  scenario:=triple_meet
```

预期行为：

三方同时接近交汇点，每台机器人独立计算优先级（假设初始 yield_count 均为 0，robot_id 字典序决定：A > B > C）：

```plain text
| 阶段 | robot_a | robot_b | robot_c |
|------|---------|---------|---------|
| 感知 | AWARENESS | AWARENESS | AWARENESS |
| 警戒 | CAUTION | CAUTION | CAUTION |
| 协商 | PASSING (30%) | YIELDING (0%) | YIELDING (0%) |
| A通过后 | NORMAL | PASSING (30%) | YIELDING (0%) |
| B通过后 | NORMAL | NORMAL | PASSING (30%) |
| C通过后 | NORMAL | NORMAL | NORMAL |
```

Pass Criteria：

- ✅ robot_a 优先级最高，减速通过

- ✅ robot_b 和 robot_c 等待

- ✅ A 通过后 B 开始通过（非同时恢复，无饿死）

- ✅ 三方依次通过，无死锁

- ✅ 每轮完整循环后，原优先级最低的 robot_c 的 yield_count +1，下次优先级提升

饿死防护验证：

```bash
# 连续观察 3 轮（每轮约 30s），验证 robot_c 的 yield_count
ros2 topic echo /robot_c/fleet/coordinator_status
# 预期：yield_count 在每轮后递增
```

失败判定：

- ❌ 任意两车同时恢复导致碰撞

- ❌ robot_c 连续 3 轮仍为最低优先级（饿死）

- ❌ 死锁（两车互相等待）

### 9.5 异常场景仿真测试

### 9.5.1 EMERGENCY 紧急停车（E2E-SIM-005）

操作：在 Gazebo 中手动拖拽 robot_b 靠近 robot_a 至距离 < 0.8m。

预期：

- 双方立即进入 EMERGENCY 状态，速度=0

- 响应时间 ≤ 200ms（从距离 < 0.8m 到 cmd_vel=0）

验证命令：

```bash
# 记录时间戳
ros2 topic echo /robot_a/fleet/coordinator_status | grep EMERGENCY
# 对比 Gazebo 中模型位置
```

### 9.5.2 通信断连降级（E2E-SIM-006）

操作：在仿真中关闭 robot_b 的 domain_bridge 进程：

```bash
# 在 robot_b 容器/进程中
killall domain_bridge
```

预期：

- robot_a 检测到 heartbeat 超时 6s

- robot_a 将 robot_b 标记为超时 peer，不再等待

- robot_a 恢复正常独立运行（不死锁）

验证：

```bash
# 观察 robot_a 日志
ros2 run rqt_console rqt_console
# 预期：出现 heartbeat timeout 警告，peer 降级
```

### 9.5.3 让行超时恢复（E2E-SIM-007）

操作：模拟 robot_a 故障（不发送 RESUME），robot_b 等待 15s 后自动恢复。

验证：

```bash

ros2 topic echo /robot_b/fleet/coordinator_status
```

### 9.7 云端部署注意事项

### 9.7.1 Gazebo 仿真资源

```plain text
| 规格 | 最低要求 | 推荐 |
|------|---------|------|
| CPU | 4 核 | 8 核 |
| 内存 | 8 GB | 16 GB |
| GPU | 无要求（Headless 模式） | NVIDIA GPU + NVIDIA driver |
| 磁盘 | 10 GB | 20 GB |
```

### 9.7.2 Headless 模式（无 GUI）

如在云端无 GPU 环境，使用 Headless 模式：

```bash
# 设置 Headless（无渲染窗口）
export GAZEB0_RENDER0_DEVICE=llvmpipe  # 软件渲染
gzserver --verbose worlds/corridor.world

# 或使用 xvfb（虚拟帧缓冲）
xvfb-run -a ros2 launch fleet_sim_gazebo simulation_2robots.launch.py
```

### 9.7.3 ROS_DOMAIN_ID 隔离

云端多容器部署时，确保 fleet topics 通过 Domain 42 隔离：

```bash
# robot_a 容器
export ROS_DOMAIN_ID=0          # 内部通信
export CYCLONEDDS_URI=file:///etc/cyclonedds/internal_sim.xml

# robot_b 容器
export ROS_DOMAIN_ID=0
export CYCLONEDDS_URI=file:///etc/cyclonedds/internal_sim.xml

# domain_bridge 在各自容器内运行，桥接 Domain 0 ↔ Domain 42
```

### 9.7.4 仿真时间加速

真实 7 天 × 8h = 168h 的 E2E 测试，可通过仿真加速：

```bash
# Gazebo 仿真时间因子（不影响物理仿真精度）
gz physics -u 10.0  # 10x 加速，168h → 16.8h

# Nav2 use_sim_time 确保所有节点使用仿真时间
ros2 launch fleet_sim_gazebo simulation_2robots.launch.py \
  use_sim_time:=true \
  sim_speed_factor:=10.0
```
